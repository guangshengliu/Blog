Red-Black Trees

1. Introduction to the Red-Black Tree

Definition of Red-Black Tree:
R-B tree is a special Red-Black Tree. Each node of the Red-Black Tree has a storage bit representing the color of the node, which can be red or black. It was invented by Rudolf Bayer in 1972, when it was called symmetric binary B-trees. Later, in 1978, it was modified by Leo J. guibas and Robert Sedgewick to the present-day Red-Black Tree. Similar to AVL tree, Red-Black Tree keeps the balance of Red-Black Tree through specific operations during insertion and deletion, so as to obtain high search performance. It can search, insert and delete in O (log n) time, where n is the number of elements in the tree.
Black tree is widely used, for example, TreeSet and treemap in Java collection, set and map in C + + STL, and management of Linux virtual memory are all realized through Red-Black Tree.

Characteristics of the Red-Black Tree:

  1. Each node is either black or red.
  2. The root node is black.
  3. Each leaf node (NIL) is black. [Note: here the leaf node refers to the empty (nil or null) leaf node! ]
  4. If a node is red, its children must be black.
  5. All paths from a node to its children contain the same number of black nodes.

3.Search for Red-Black tree

Because Red-Black tree is a special Red-Black Tree, the search method of Red-Black tree is basically the same as that of Red-Black Tree.
Recursive and non recursive search codes are as follows:

// (recursive implementation) find the node whose key value is key in the Red-Black Tree.
template <class T>
RBTNode<T>* RBTree<T>::search(RBTNode<T>* x, T key) const
{
    if (x==NULL || x->key==key)
        return x;

    if (key < x->key)
        return search(x->left, key);
    else
        return search(x->right, key);
}
template <class T>
RBTNode<T>* RBTree<T>::search(T key) 
{
    search(mRoot, key);
}

//	(non recursive implementation) find the node whose key value is key in the Red-Black Tree.
template <class T>
RBTNode<T>* RBTree<T>::iterativeSearch(RBTNode<T>* x, T key) const
{
    while ((x!=NULL) && (x->key!=key))
    {
        if (key < x->key)
            x = x->left;
        else
            x = x->right;
    }

    return x;
}

template <class T>
RBTNode<T>* RBTree<T>::iterativeSearch(T key)
{
    iterativeSearch(mRoot, key);
}

4.Left and right rotations of Red-Black tree

The basic operations of Red-Black Tree are adding and deleting. The rotation method is used after adding or deleting the Red-Black Tree. Why? The reason is very simple. After adding or deleting the nodes in the red black tree, the Red-Black Tree changes. It may not meet the five properties of the Red-Black Tree, and it is no longer a Red-Black Tree, but a common tree. By rotating, the tree can become a Red-Black Tree again. In a nutshell, the purpose of rotation is to keep the tree as red black. There are two types of rotation: left and right. They are described below.
The left-hand adjustment procedure is as follows:

  1. Let the left subtree of B become the right subtree of A.
  2. Let A be the left subtree of B.
  3. Set root node as node B.

According to the above ideas, the code is as follows:

//	Left rotation
template <class T>
void RBTree<T>::leftRotate(RBTNode<T>* &root, RBTNode<T>* x)
{
    // Set the right child of X to Y.
    RBTNode<T> *y = x->right;
    // Set Y's left child to X's right child.
    // If y's left child is not empty, set X as the father of Y's left child.
    x->right = y->left;
    if (y->left != NULL)
        y->left->parent = x;
    // Set father of X as father of Y.
    y->parent = x->parent;
    if (x->parent == NULL)
    {
        root = y;            // If the father of X is an empty node, y is set as the root node.
    }
    else
    {
        if (x->parent->left == x)
            x->parent->left = y;    // If X is the left child of its parent, set Y as the left child of its parent.
        else
            x->parent->right = y;    // If X is the left child of its parent, set Y as the left child of its parent. 
    }    
    // Left child with X set to Y
    y->left = x;
    //Set X's parent to y
    x->parent = y;
}

Since there is a left turn, there must be a right turn. In fact, right-hand and left-hand rotation are symmetrical processes, so left-hand reverse operation can be carried out.
Right turn adjustment steps are as follows:

  1. Let the right subtree of A become the left subtree of B.
  2. Let B be the left subtree of A.
  3. Set root node as node A.

According to the above ideas, the code is as follows:

//	Right rotation
template <class T>
void RBTree<T>::rightRotate(RBTNode<T>* &root, RBTNode<T>* y)
{
    // Set X to be the left child of the current node.
    RBTNode<T> *x = y->left;
    // Set the right child of X as the left child of Y.
    // If the right child of X is not empty, set Y as the father of the right child of X.
    y->left = x->right;
    if (x->right != NULL)
        x->right->parent = y;
    // Make Y's father X's father
    x->parent = y->parent;

    if (y->parent == NULL) 
    {
        root = x;            // If the father of Y is an empty node, X is set as the root node
    }
    else
    {
        if (y == y->parent->right)
            y->parent->right = x;    // If y is the right child of its parent, set X as the right child of Y's parent
        else
            y->parent->left = x;    // (y is the left child of its parent node) the left child of the parent node that sets x as X
    }
    // Right child with Y as X
    x->right = y;
    // Set Y's parent to X
    y->parent = x;
}

5.Insertion of Red-Black tree

What steps do you need to take to insert a node into the Red-Black Tree? First, the Red-Black Tree is regarded as a Red-Black Tree, and the nodes are inserted; then, the nodes are colored red; finally, the tree is modified by rotation and recolor to make it a Red-Black Tree again. The details are as follows:

Step 1: take the Red-Black Tree as a Red-Black Tree, and insert the nodes.

The Red-Black Tree itself is a Red-Black Tree. After the nodes are inserted, the tree is still a binary search tree. This means that the key values of the tree are still in order. In addition, whether it is left-handed or right-handed, if the tree is a Red-Black Tree before rotation, it must be a binary search tree after rotation. This means that any rotation and recolor operation will not change the fact that it is still a Red-Black Tree. OK Then, we will try our best to rotate and recolor the tree to make it a red black tree again!

Step 2: color the inserted nodes red.

Why is it colored red instead of black? Why? Before answering, we need to review the characteristics of red black trees:

  1. Each node is either black or red.
  2. The root node is black.
  3. Each leaf node is black. [Note: here the leaf node refers to the empty leaf node! ]
  4. If a node is red, its children must be black.
  5. All paths from a node to its children contain the same number of black nodes.

Coloring the inserted node red will not violate the feature (5) and one less violation means that we need to deal with fewer situations. Next, we should try our best to make this tree satisfy other properties. If it is satisfied, it will be a Red-Black Tree again.

Step 3: through a series of rotation or coloring operations, make it a Red-Black Tree again.

In the second step, after the insertion node is colored red, the property (5) is not violated. What characteristics does it violate?
feature (1), it is obvious that it will not be violated. Because we have painted it red.
For feature (2), it is clear that it will not be violated. In the first step, we treat the Red-Black Tree as a Red-Black Tree, and then insert it. According to the characteristics of binary lookups, the insertion does not change the root node. So the root node is still black.
For feature (3), it is clear that it will not be violated. The leaf node here refers to the empty leaf node. Inserting non empty nodes will not affect them.
For feature (4), it is possible to violate! Then, find a way to satisfy the feature (4), and you can reconstruct the tree into a Red-Black Tree.
According to the case of the parent node of the inserted node, when the node Z is colored as a red node, and the inserted binary tree is divided into three cases.

  1. Description: the inserted node is the root node.Treatment method: paint this node black directly.
  2. Description: the parent node of the inserted node is black. Treatment: nothing needs to be done. After the node is inserted, it is still a Red-Black Tree.
  3. Description: the parent node of the inserted node is red. Processing method: then, this situation conflicts with the feature (5) of the Red-Black Tree. In this case, there must be a non empty grandfather node in the inserted node; furthermore, there must be an uncle node in the inserted node (even if the uncle node is empty, we also think it exists, and the empty node itself is a black node). After understanding this, we further divide this situation into three cases according to the situation of Uncle node.
Phenomenon description Processing strategy
Case 1 The parent node of the current node is red, and another child node (Uncle node) of the grandfather node of the current node is also red. (1) set the parent node to black.
(2) set uncle node to black.
(3) set grandfather node to red.
(4) set grandfather to current (red); that is, continue working on current.
Case 2 The parent node of the current node is red, the uncle node is black, and the current node is the right child of its parent node. (1) use parent as the new current node.
(2) turn left with new current node as the fulcrum.
Case 3 The parent node of the current node is red, the uncle node is black, and the current node is the left child of its parent node. (1) set the parent node to black.
(2) set grandfather node to red.
(3) turn right with grandfather node as the fulcrum.

The core idea of the above three cases is to move the red node to the root node, and then set the root node to black. They are described in detail below.

(case 1) Uncle node is red

1. phenomenon description

The parent node of the current node (that is, the inserted node) is red, and the other child node (Uncle node) of the grandfather node of the current node is also red.

1.2 handling strategy

(1) set the parent node to black.
(2) set the uncle node to black.
(3) set the grandfather node to red.
(4) set the grandfather node as the current node (red node); that is, continue to operate on the current node later.

Let's talk about why we should do this. (It is recommended to compare the following figures when understanding)

Both the current node and the parent node are red, violating the attribute (4). Therefore, set the parent node to black to solve this problem. However, changing the parent from red to black violates the property (5): because the total number of black nodes that contain branches of the parent increases by 1. The solution to this problem is to change the grandfather node from black to red, and the uncle node from red to black. Here are some points:

First, why is "grandfather node" black before? This should be easy to understand, because before the transformation operation, the tree is a Red-Black Tree, and the "parent node" is red, so the "grandfather node" must be black.

Second, why change the grandfather node from black to red, and at the same time, change the uncle node from red to black? It can solve the problem that the total number of black nodes including the branches of the parent node increases by 1. It's a very simple truth. The total number of black nodes with branches of the parent node increased by 1.
At the same time, it also means that the total number of black nodes in the branch containing the grandfather node increases by 1. In this case, we can solve the problem that the total number of black nodes in the branch containing the grandfather node increases by 1 by changing the grandfather node from black to red.
However, after this processing, another problem will be caused. The total number of black nodes including the branches of Uncle nodes has been reduced by 1. Now we know that uncle nodes are red. Setting uncle nodes to black can solve this problem.
Therefore, the problem is solved by changing the grandfather node from black to red and the uncle node from red to black.

After following the above steps: the current node, the parent node and the uncle node will not violate the red black tree feature, but the grandfather node is not necessarily. At this time, if the grandfather node is the root node and the grandfather node is set to black directly, this problem will be completely solved; if the grandfather node is not the root node, then we need to set the grandfather node as the new current node, and then analyze the new current node.

(1) make the parent node the new current node.

Let's talk about why we should do this. (It is recommended to compare the following figures when understanding)

First, the parent node is used as the new current node; then, the new current node is used as the fulcrum to rotate left.For ease of understanding, we first describe step (2) and then step (1); for ease of description, we set the code of the parent node as f (father) and the code of the current node as s (son).

Why do we need to rotate left with F as the fulcrum? According to the known conditions, s is the right child of F. As we said before, we deal with the core idea of the Red-Black Tree: move the red node to the root node; then, set the root node to black. Since the red node is moved to the root node, that is to say, the red node that destroys the properties of the Red-Black Tree should be continuously moved up (that is, to the root). And S is a right child, so we can move S up by turning left!

After processing according to the above steps (left turning with F as the fulcrum): if s becomes the root node, set it to black directly to completely solve the problem; if s is not the root node, then we need to perform step (1), that is, set F as "new current node". Why not continue processing with s as the new current node and F as the new current node? This is because after left-handed rotation, f becomes the child node of S, that is, s becomes the parent node of F. when we deal with the problem, we need to deal with it from the bottom to the top (from leaf to root). That is to say, we must solve the problem of the child first, and then the problem of the father. Therefore, we perform step (1): take the parent node as the new current node.

We can change f from red to black, which solves the problem of violating the property (4); however, it causes other problems: violating the property (5), because after changing f from red to black, the number of black nodes in all branches of F increases by 1. So how can we solve the problem that the number of black nodes passing through all branches of F increases by 1? We can solve this problem by turning G from black to red, and turning g right at the same time.

//	Red-Black Tree insertion correction function
//	Call this function after inserting a node (out of balance) into the Red-Black Tree; the purpose is to reshape it into a Red-Black Tree.
template <class T>
void RBTree<T>::insertFixUp(RBTNode<T>* &root, RBTNode<T>* node)
{
    RBTNode<T> *parent, *gparent;

    // If the parent node exists, and the color of the parent node is red
    while ((parent = rb_parent(node)) && rb_is_red(parent))
    {
        gparent = rb_parent(parent);

        //If the parent node is the left child of the grandfather node
        if (parent == gparent->left)
        {
            // Case 1 condition: Uncle node is red
            {
                RBTNode<T> *uncle = gparent->right;
                if (uncle && rb_is_red(uncle))
                {
                    rb_set_black(uncle);
                    rb_set_black(parent);
                    rb_set_red(gparent);
                    node = gparent;
                    continue;
                }
            }

            // Case 2 condition: uncle is black, and the current node is the right child.
            if (parent->right == node)
            {
                RBTNode<T> *tmp;
                leftRotate(root, parent);
                tmp = parent;
                parent = node;
                node = tmp;
            }

            // Case 3 condition: the uncle is black, and the current node is the left child.
            rb_set_black(parent);
            rb_set_red(gparent);
            rightRotate(root, gparent);
        } 
        else//If Z's parent is Z's grandfather's right child.
        {
            // Case 1 condition: Uncle node is red.
            {
                RBTNode<T> *uncle = gparent->left;
                if (uncle && rb_is_red(uncle))
                {
                    rb_set_black(uncle);
                    rb_set_black(parent);
                    rb_set_red(gparent);
                    node = gparent;
                    continue;
                }
            }

            // Case 2 condition: uncle is black, and the current node is the left child.
            if (parent->left == node)
            {
                RBTNode<T> *tmp;
                rightRotate(root, parent);
                tmp = parent;
                parent = node;
                node = tmp;
            }

            // Case 3 condition: the uncle is black, and the current node is the right child.
            rb_set_black(parent);
            rb_set_red(gparent);
            leftRotate(root, gparent);
        }
    }

    // Set the root node to black.
    rb_set_black(root);
}
//	Find the successor node of node (x). That is, the data value in the Red-Black Tree is greater than the minimum node of the node.
template <class T>
RBTNode<T>* RBTree<T>::successor(RBTNode<T> *x)
{
    //  If x has a right child, the successor node of X is the smallest node of the subtree whose root is the right child.
    if (x->right != NULL)
        return minimum(x->right);

    // If x has no right child. Then x has the following two possibilities:
	//(01) x is a left child, then the successor node of X is its parent node.
	//(02) if x is a right child, find the lowest parent node of X, and the parent node must have a left child. The lowest parent node found is the successor node of X.
    RBTNode<T>* y = x->parent;
    while ((y!=NULL) && (x==y->right))
    {
        x = y;
        y = y->parent;
    }

    return y;
} 
// Find the precursor node of node (x). That is, the data value in the Red-Black Tree is less than the maximum node of the node.
template <class T>
RBTNode<T>* RBTree<T>::predecessor(RBTNode<T> *x)
{
    // If x has a left child, the leading node of X is the largest node of the subtree whose root is the left child.
    if (x->left != NULL)
        return maximum(x->left);
    // If x doesn't have a left child. Then x has the following two possibilities:            
	//(01) x is a right child, then the precursor node of X "is its parent node.            
	//(01) if x is a left child, find the lowest parent node of X, and the parent node must have a right child. The "lowest parent node" found is the precursor node of X.
    RBTNode<T>* y = x->parent;
    while ((y!=NULL) && (x==y->left))
    {
        x = y;
        y = y->parent;
    }

    return y;
}
//	Insert node into Red-Black Tree.
template <class T>
void RBTree<T>::insert(RBTNode<T>* &root, RBTNode<T>* node)
{
    RBTNode<T> *y = NULL;
    RBTNode<T> *x = root;

    // 1. Take the Red-Black Tree as a Red-Black Tree, and add the nodes to the Red-Black Tree.
    while (x != NULL)
    {
        y = x;
        if (node->key < x->key)
            x = x->left;
        else
            x = x->right;
    }

    node->parent = y;
    if (y!=NULL)
    {
        if (node->key < y->key)
            y->left = node;
        else
            y->right = node;
    }
    else
        root = node;

    // 2. Set the node color to red.
    node->color = RED;

    // 3. Revise it to a Red-Black Tree.
    insertFixUp(root, node);
}
// Insert the node (key is the node key value) into the Red-Black Tree.
template <class T>
void RBTree<T>::insert(T key)
{
    RBTNode<T> *z=NULL;

    // If the new node fails, return.
    if ((z=new RBTNode<T>(key,BLACK,NULL,NULL,NULL)) == NULL)
        return ;

    insert(mRoot, z);
}

6.Delete the Red-Black tree

Delete a node in the Red-Black Tree. The operations to be performed are as follows: first, the Red-Black Tree is treated as a Red-Black Tree, and the node is deleted from the Red-Black Tree; then, the tree is modified to become a Red-Black Tree again by rotating and recolor. The details are as follows:

Step 1: take the Red-Black Tree as a Red-Black Tree and delete the nodes.
This is the same as deleting nodes in a regular Red-Black Tree. There are three situations:

  1. If the deleted node has no son, it is a leaf node. Then, it is OK to delete the node directly.
  2. The deleted node has only one son. Then, delete the node directly and replace its location with its unique child node.
  3. The deleted node has two sons. Then, first find out its successor node; then copy the content of its successor node to the content of the node; then, delete its successor node. In this case, the successor node is equivalent to the avatar. After the content of the successor node is copied to the deleted node, the successor node is deleted. In this way, the problem is cleverly transformed into the case of deleting subsequent nodes. Next, consider the subsequent nodes. When the deleted node has two non empty child nodes, its subsequent nodes can not be Gemini non empty. Since the subsequent nodes of the node cannot be both non empty, it means that the subsequent nodes of the node either have no sons or have only one son. If there is no son, it shall be handled according to case 1; if there is only one son, it shall be handled according to case 2.

Step 2: modify the tree to make it a Red-Black Tree by rotating and recolor.
Because after deleting the node in the first step, it may violate the characteristics of the Red-Black Tree. So you need to rotate and recolor the tree to make it a Red-Black Tree again.

In the first step, the Red-Black Tree is treated as a Red-Black Tree. After the node is deleted, three characteristics (2), (4) and (5) may be violated. The second step needs to solve the above three problems, so as to maintain all the characteristics of the Red-Black Tree.
For analysis purposes, let's assume that x contains an extra black color (x's original color still exists), so that the feature (5) is not violated. Why?

Through rb-delete algorithm, we know that after deleting node y, X occupies the position of the original node y. Since removing y (y is black) means reducing a black node, then adding a black node to the position is enough. In this way, when we assume that x contains an extra black, we can make up for the loss of the black node by deleting y, and we will not violate the feature (5). Therefore, suppose that x contains an extra black color (x's original color still exists), so that the feature (5) is not violated. Now, X contains not only its original color attribute, but also an extra black. That is, the color attribute of X is red + black or black + black, which violates the property (1).

Now, the problem we are facing has been transformed from solving the three characteristics of violation (2), (4) and (5) to solving the three characteristics of violation (1), (2) and (4). What rb-delete-fixup needs to do is to restore the properties (1), (2) and (4) of the Red-Black Tree through the algorithm. Rb-delete-fixup's idea is: move the extra black contained in x up the tree (towards the root) until the following posture appears:

A) x points to a red + Black node. In this case, set X as a black node.
B) x points to the root. In this case, set X as a black node.
C) not the first two.

The above posture can be summarized into three situations.

  1. Description: X is red + Black node.
    How to deal with it: set X to black directly and finish. At this time, all the properties of mangrove were recovered.
  2. Description: X is black + Black node, and X is root.
    How to deal with it: do nothing, it's over. At this time, all the properties of mangrove were recovered.
  3. Description: X is black + Black node, and X is not root.
    Treatment method: this kind of situation can be divided into 4 seed situations. The four seeds are shown in the table below:
Phenomenon description Processing strategy

(2) set the parent of X to red.
(3) rotate the parent node of X to the left.

(2) Set the parent of X as the new x node.

1.1 phenomenon description

X is black + Black node, and X's brother node is red. (in this case, the parent node of X and the child node of the sibling node of X are both black nodes.)

1.2 handling strategy

(1) set the brother node of X to black.
(2) set the parent of X to red.
(3) rotate the parent node of X to the left.
(4) after turning left, reset X's sibling node.

Let's talk about why we should do this. (It is recommended to compare the following figures when understanding)

The purpose of this is to convert case 1 to case 2, case 3, or case 4 for further processing. The parent node of X is rotated to the left; after the left, in order to maintain the Red-Black Tree characteristics, the sibling node of X needs to be set to black before the left rotation, and the parent node of X needs to be set to red at the same time; after the left rotation, because the sibling node of X has changed, the sibling node of X needs to be updated for subsequent processing.

(case 2) X is black + black node, X's brother node is black, X's brother node's two children are black

2.1 phenomenon description

X is black + Black node, X's brother node is black, X's brother node's two children are black.

2.2 handling strategy

(1) set the brother node of X to red.
(2) set the parent node of X as the new x node.
Let's talk about why we should do this. (it is recommended to compare the following figures when understanding)

The idea of this case is to move up (toward the root) one of the redundant black attributes in X. X is a black + Black node. We change x from a black + Black node to a black node. The extra black attribute is moved to X's parent node, that is, X's parent node has an extra black attribute (if x's parent node was originally black, then it becomes black + black; if X's parent node was originally red, then it becomes red + Black). At this time, it should be noted that the number of black nodes in all branches passing X has not changed; however, the number of black nodes in all branches passing X's sibling nodes has increased by 1 (because X's parent node has an additional black attribute)! In order to solve this problem, we need to reduce the number of black nodes in all branches of X's sibling nodes by 1. Then we can change X's sibling nodes from black to red.

After the above steps (set X's sibling node to red), an extra color attribute (black) has run to X's parent node. We need to set the parent of X as a new x node for processing. If the new x node is black + red, the problem can be solved by setting the new x node to black directly; if the new x node is black + black, the new x node needs to be further processed.

X is black + Black node, X's brother node is black; X's brother node's left child is red, and the right child is black.

3.2 handling strategy

(1) set the left child of the X brother node to black.
(2) set the X sibling node to red.
(3) right turn the sibling node of X.
(4) after turning right, reset X's sibling node.

Let's talk about why we should do this. (it is recommended to compare the following figures when understanding)

The purpose of case 3 is to transform case 3 into case 4 for further processing. The way of conversion is to rotate the brother node of X to the right; to ensure that it is still a Red-Black Tree after the right rotation, the left child of the brother node of X needs to be set to black before the right rotation, and the brother node of X needs to be set to red at the same time; after the right rotation, because the brother node of X has changed, the brother node of X needs to be updated for subsequent processing.

3.3 schematic diagram

(case 4) x is black + black node, X's brother node is black; X's brother node's right child is red, X's brother node's left child is any color

4.1 phenomenon description

X is black + Black node, X's brother node is black; X's brother node's right child is red, and X's brother node's left child is any color.

4.2 handling strategy

(1) assign x parent node color to X's siblings. (2) set the X parent node to black. (3) set the right sub section of the X sibling node to black. (4) rotate the parent node of X to the left. (5) set X as the root node.

Let's talk about why we should do this. (It is recommended to compare the following figures when understanding)

The purpose of case 4 is to remove the extra black in X and change x to a separate black. The way to deal with this is to make a color change, and then rotate the X's parent node to the left. Next, let's analyze how it is implemented.

For ease of explanation, we set the current node as s (original son), brother node as B (brother), left child of brother node as BLS (brother & ා s leftson), right child of brother node as BRS (brother & ා s right son), and parent node as f (father).

We're going to turn f left. But before turning left, we need to change the colors of F and B and set BRS to black. Why do you need to deal with it here? Since F and BLS are parent-child relationships after left rotation, we know that BL is red. If f is red, it violates the feature (4); to solve this problem, we set F to black. However, after f is set to black, in order to ensure that the feature (5) is met, i.e. to ensure that after left turning:

First, the number of black nodes passing through both the root node and the branch of s does not change.

If it meets the first requirement, it only needs s to discard its redundant colors. Because the color of S is black + black, and the number of black nodes passing through both the root node and the branch of s after left-handed rotation has increased by 1; now, only changing s from black + black to a separate black node can meet the first requirement.

Second, the number of black nodes passing through both the root node and the BLS branch remains the same.

If the second is satisfied, only the original color of F needs to be assigned to B. Previously, we have set F to black (that is, the color black of B is assigned to f). So far, we've changed the colors of F and B.

Third, the number of black nodes passing through both the root node and the BRS branch does not change.

In case the second has been met, to meet the third, just set BRS to black. After, after the above processing. All the characteristics of the Red-Black Tree are satisfied! Next, we set X as the root node, and we can jump out of the while loop (refer to the pseudo code); that is, we have completed all the processing.

At this point, we have finished processing case 4. The core of understanding case 4 is to understand how to remove the extra black of the current node.

//This function is called after the insertion node is removed from the Red-Black Tree (the Red-Black Tree is out of balance); 
//the purpose is to reshape it into a Red-Black Tree.
template <class T>
void RBTree<T>::removeFixUp(RBTNode<T>* &root, RBTNode<T> *node, RBTNode<T> *parent)
{
    RBTNode<T> *other;

    while ((!node || rb_is_black(node)) && node != root)
    {
        if (parent->left == node)
        {
            other = parent->right;
            if (rb_is_red(other))
            {
                // Case 1: brother w of X is red
                rb_set_black(other);
                rb_set_red(parent);
                leftRotate(root, parent);
                other = parent->right;
            }
            if ((!other->left || rb_is_black(other->left)) &&
                (!other->right || rb_is_black(other->right)))
            {
                // Case 2: brother w of X is black, and both children of W are black 
                rb_set_red(other);
                node = parent;
                parent = rb_parent(node);
            }
            else
            {
                if (!other->right || rb_is_black(other->right))
                {
                    // Case 3: brother w of X is black, and the left child of W is red, and the right child is black.
                    rb_set_black(other->left);
                    rb_set_red(other);
                    rightRotate(root, other);
                    other = parent->right;
                }
                // Case 4: brother w of X is black, and the right child of W is red, and the left child is any color.
                rb_set_color(other, rb_color(parent));
                rb_set_black(parent);
                rb_set_black(other->right);
                leftRotate(root, parent);
                node = root;
                break;
            }
        }
        else
        {
            other = parent->left;
            if (rb_is_red(other))
            {
                // Case 1: brother w of X is red  
                rb_set_black(other);
                rb_set_red(parent);
                rightRotate(root, parent);
                other = parent->left;
            }
            if ((!other->left || rb_is_black(other->left)) &&
                (!other->right || rb_is_black(other->right)))
            {
                // Case 2: brother w of X is black, and both children of W are black
                rb_set_red(other);
                node = parent;
                parent = rb_parent(node);
            }
            else
            {
                if (!other->left || rb_is_black(other->left))
                {
                    // Case 3: brother w of X is black, and the left child of W is red, and the right child is black.  
                    rb_set_black(other->right);
                    rb_set_red(other);
                    leftRotate(root, other);
                    other = parent->left;
                }
                // Case 4: brother w of X is black, and the right child of W is red, and the left child is any color.
                rb_set_color(other, rb_color(parent));
                rb_set_black(parent);
                rb_set_black(other->left);
                rightRotate(root, parent);
                node = root;
                break;
            }
        }
    }
    if (node)
        rb_set_black(node);
}

// Delete node and return the deleted node.
template <class T>
void RBTree<T>::remove(RBTNode<T>* &root, RBTNode<T> *node)
{
    RBTNode<T> *child, *parent;
    RBTColor color;

    // The left and right children of the deleted node are not empty.
    if ( (node->left!=NULL) && (node->right!=NULL) ) 
    {
        // The successor node of the deleted node. (called replacement node)
        // Use it to replace the "deleted node" position, and then remove the deleted node.
        RBTNode<T> *replace = node;

        // Get successor node
        replace = replace->right;
        while (replace->left != NULL)
            replace = replace->left;

        // Node node is not a root node (only the root node does not have a parent node)
        if (rb_parent(node))
        {
            if (rb_parent(node)->left == node)
                rb_parent(node)->left = replace;
            else
                rb_parent(node)->right = replace;
        } 
        else 
            // Node node is the root node, update the root node.
            root = replace;

        // Child is the right child to replace the node, and also the node to be adjusted.
        // There must be no left child in the replacement node! Because it is a successor node.
        child = replace->right;
        parent = rb_parent(replace);
        // Save the color of the replacement node.
        color = rb_color(replace);

        // The deleted node is the parent of its successor.
        if (parent == node)
        {
            parent = replace;
        } 
        else
        {
            // Child is not empty.
            if (child)
                rb_set_parent(child, parent);
            parent->left = child;

            replace->right = node->right;
            rb_set_parent(node->right, replace);
        }

        replace->parent = node->parent;
        replace->color = node->color;
        replace->left = node->left;
        node->left->parent = replace;

        if (color == BLACK)
            removeFixUp(root, child, parent);

        delete node;
        return ;
    }
    if (node->left !=NULL)
        child = node->left;
    else 
        child = node->right;
    parent = node->parent;
    // Save the color of the replacement node.
    color = node->color;
    if (child)
        child->parent = parent;
    // Node node is not the root node
    if (parent)
    {
        if (parent->left == node)
            parent->left = child;
        else
            parent->right = child;
    }
    else
        root = child;

    if (color == BLACK)
        removeFixUp(root, child, parent);
    delete node;
}

// Delete the node whose key value is key in the Red-Black Tree.
template <class T>
void RBTree<T>::remove(T key)
{
    RBTNode<T> *node; 

    // Find the node corresponding to the key, and delete the node if found.
    if ((node = search(mRoot, key)) != NULL)
        remove(mRoot, node);
}

10. Traversal of Red-Black Tree

(1) first order traversal

For the first order traversal, the root node root is always visited first, and then the left and right subtrees are visited. Therefore, the traversal order of the first order traversal is root node -> left subtree -> right subtree. From this, you can write the code of traversal in sequence:

//	Preorder traversal of Red-Black Tree.
template <class T>
void RBTree<T>::preOrder(RBTNode<T>* tree) const
{
    if(tree != NULL)
    {
        cout<< tree->key << " " ;
        preOrder(tree->left);
        preOrder(tree->right);
    }
}
template <class T>
void RBTree<T>::preOrder() 
{
    preOrder(mRoot);
}

(2) middle order traversal

For the middle order traversal, it always accesses the left subtree first, then the root node, and finally the right subtree. Therefore, the traversal order of the middle order traversal is left subtree -> root node -> right subtree.For the Red-Black Tree, the result is ordered. From this, you can write the code of middle order traversal:

//	Inorder traversal of Red-Black Tree.
template <class T>
void RBTree<T>::inOrder(RBTNode<T>* tree) const
{
    if(tree != NULL)
    {
        inOrder(tree->left);
        cout<< tree->key << " " ;
        inOrder(tree->right);
    }
}
template <class T>
void RBTree<T>::inOrder() 
{
    inOrder(mRoot);
}

(3) post order traversal

For post order traversal, the left subtree is always visited first, then the right subtree, and finally the root node. Therefore, the order of post order traversal is left subtree -> right subtree -> root node.
From this, you can write the code of the subsequent traversal:

//Postorder traversal of Red-Black Tree.
template <class T>
void RBTree<T>::postOrder(RBTNode<T>* tree) const
{
    if(tree != NULL)
    {
        postOrder(tree->left);
        postOrder(tree->right);
        cout<< tree->key << " " ;
    }
}
template <class T>
void RBTree<T>::postOrder() 
{
    postOrder(mRoot);
}
